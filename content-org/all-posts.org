** The unseen hero of OpenBSD: OpenBSD's malloc

*** What this is about
This is me working through OpenBSD's malloc in depth.

Not a surface-level overview.

I want to understand the internals — the data structures, the design decisions,
and why those decisions make heap exploitation so much harder.

If you're reading this: the goal was to understand it well enough to explain it
to someone I respect.

-----

*** Start here: what malloc actually does
Every C program that needs memory at runtime calls =malloc=.

=malloc= is a library function. It's not a syscall — it's a layer between your
code and the kernel.

When you write:

#+begin_src c
char *buf = malloc(64);
#+end_src

...you're asking the allocator to find 64 bytes somewhere, hand you a pointer,
and track that those bytes are in use.

When you call =free(buf)=, you're telling the allocator those bytes are available
again.

That's the contract. The allocator manages that contract.

The question is: what happens when the contract is violated?

A buffer overflow writes past the end of =buf=.
A use-after-free reads from =buf= after it's been freed.
A double free calls =free(buf)= twice.

With a naive allocator, these bugs are often silent. The program keeps running
with corrupted state. That corrupted state is what attackers exploit.

OpenBSD's malloc is designed to make these bugs loud — to turn silent corruption
into immediate, reproducible crashes.

-----

*** A brief history: how we got here

**** The original: sbrk() and one big heap
Early Unix allocators used =sbrk()= — a syscall that extends the process's data
segment upward.

Think of it as a stack of memory growing in one direction.

All allocations lived in one contiguous block. Predictable layout.
Fast. And a security problem, because attackers could reason about where
things would be in memory.

**** 2001: mmap instead of sbrk
Thierry Deval rewrote OpenBSD's malloc to use =mmap()= instead.

=mmap()= is a syscall that requests a fresh page of memory from the kernel.
Unlike =sbrk()=, it doesn't have to extend a single contiguous block.
Each call can land anywhere in the address space.

This was the first major break from the "one big heap" model.

**** 2008: Otto Moerbeek's rewrite
Otto Moerbeek did a near-complete redesign.

This is the allocator OpenBSD ships today. It's called "otto-malloc" informally.

The focus: safety, randomness, metadata integrity, and defined failure behavior.
Not performance. Safety.

**** After 2008: continued hardening
The design didn't freeze in 2008. Relevant additions since then:

- Chunk canaries — small values that detect when you've written past the end of
  a small allocation
- Delayed free lists — freed chunks are held in a queue and checked for
  double-frees before being recycled
- Use-after-free protection for large allocations — freed pages get their
  permissions set to =PROT_NONE= (no read, no write, no execute), so any access
  through a dangling pointer faults immediately
- Per-thread pools — each thread gets its own bookkeeping structure, avoiding
  lock contention and reducing cross-thread information leakage
- =malloc_readonly= in a read-only mapping — the global configuration becomes
  unwritable after the dynamic linker finishes, so you can't corrupt it through
  a write primitive

-----

*** The internal structure
This is the part I actually wanted to understand.

Here's how the allocator is organized.

**** Everything starts with =struct dir_info=
Every malloc pool is represented by one =struct dir_info=.

"Pool" here means a collection of memory managed together — one per thread in
multi-threaded programs.

=dir_info= is the central bookkeeping structure. It tracks:
- Where all the allocated regions are
- Which small-allocation slots are free
- The delayed-free queue
- A buffer of random bytes used for randomizing slot selection
- Two canary values that sandwich the struct

Here's the relevant part of the actual source (=malloc.c=, current as of 2025):

#+begin_src c
struct dir_info {
    u_int32_t canary1;         /* integrity check — start */
    int active;                /* is this pool in use? */
    struct region_info *r;     /* table of all allocated regions */
    size_t regions_total;      /* how many slots in that table */
    size_t regions_free;       /* how many are empty */
    int malloc_junk;           /* junk fill enabled? */
    int mmap_flag;             /* extra mmap flags */

    /* free lists for small allocations, one per size class */
    struct chunk_info_list[BUCKETS + 1];

    /* lists of chunk pages that still have free slots */
    struct chunk_head chunk_dir[BUCKETS + 1][MALLOC_CHUNK_LISTS];

    /* recently freed small chunks, held for double-free checking */
    void *delayed_chunks[MALLOC_DELAYED_CHUNK_MASK + 1];

    u_char rbytes[32];         /* random bytes for slot selection */
    /* ... */
    u_int32_t canary2;         /* integrity check — end */
};
#+end_src

The canaries (=canary1= and =canary2=) are the first and last fields.

If anything corrupts =dir_info= — for example, by overflowing a buffer that
happens to be near it in memory — the integrity check fires.

The allocator calls =wrterror()= and aborts the process.

This is intentional. Loud failure over silent corruption.

**** The global config lives in read-only memory
The global configuration — guard page size, junk level, the pool pointers — is
stored in =struct malloc_readonly=:

#+begin_src c
struct malloc_readonly {
    struct dir_info *malloc_pool[_MALLOC_MUTEXES]; /* one pool per thread */
    u_int malloc_mutexes;      /* how many pools are active */
    /* malloc_guard, malloc_junk, etc. */
    u_int32_t malloc_canary;   /* cross-checked against each pool's canary1 */
};

static struct malloc_readonly mopts __relro;
#+end_src

=__relro= means "relocations read-only". After the dynamic linker finishes
setting up the process, this section becomes read-only at the hardware level.

Any write to =mopts= through a bug or exploit primitive hits a read-only page
and faults immediately.

You can't corrupt the allocator's configuration without crashing the process
first.

**** The metadata is not next to your data
This is the key architectural decision.

In glibc's allocator, chunk headers sit immediately before user data.
The metadata describing an allocation lives at =(ptr - 8)= or similar.

If you overflow your buffer, you can overwrite that metadata.
Classic heap exploits are built on exactly this.

In otto-malloc, =dir_info= and =chunk_info= live in completely separate =mmap=
regions. There is no chunk header adjacent to user data.

Running past the end of your buffer hits either another allocation, a guard page,
or unmapped memory. Not metadata.

**** Small allocations: chunks and buckets
Allocations smaller than half a page go into chunk pages.

A chunk page is one =mmap='d page divided into uniform slots of the same size.

Size classes run from 16 bytes (=MALLOC_MINSIZE=) up to half a page
(=MALLOC_MAXCHUNK=), in powers of two. Each size class is a "bucket".

Each chunk page is described by a =struct chunk_info=:

#+begin_src c
struct chunk_info {
    LIST_ENTRY(chunk_info) entries; /* linked list of chunk pages */
    void *page;          /* pointer to the actual page */
    u_short canary;      /* boundary integrity value */
    u_short bucket;      /* which size class (16b, 32b, 64b...) */
    u_short free;        /* how many slots are still free */
    u_short total;       /* total number of slots on this page */
    u_short offset;      /* size table offset */
    u_short bits[...];   /* bitmap: 1 = slot is free */
};
#+end_src

The =canary= here is not a stack canary. It's a per-chunk-page value that gets
checked when a slot is freed. If the boundary between slots has been written
past, the check fails and the process aborts.

Slot selection within a chunk page uses the =rbytes= pool from =dir_info=.

Which specific slot you get from =malloc= is not deterministic. Two consecutive
calls to =malloc(64)= don't necessarily give you adjacent addresses.

This breaks a core assumption that heap grooming relies on: predictable relative
placement between allocations.

**** Large allocations: their own mmap region
Allocations at or above one page get their own dedicated =mmap= region.

They're tracked as entries in the region table (=dir_info->r=), not in any chunk
structure.

When freed, they can go back to the kernel via =munmap=. Any dangling pointer
to that address will fault on the next access — the page no longer exists in the
process's address space.

The guard page (when enabled) is an unmapped page appended immediately after the
allocation:

#+begin_src c
#define MALLOC_GUARD ((size_t)MALLOC_PAGESIZE)
#+end_src

One byte past the end of your allocation hits this page. Immediate fault.

**** The junk fill values
Two specific byte patterns matter here:

#+begin_src c
#define SOME_JUNK      0xdb  /* "deadbeef" — written on fresh allocation */
#define SOME_FREEJUNK  0xdf  /* "dead, free" — written before free */
#+end_src

=0xdb= on allocation makes reading uninitialized memory produce garbage that
almost always causes a crash or detectable bad state.

=0xdf= on free marks the region as dead. Any use-after-free that reads from this
region will get =0xdf= bytes back. That rarely looks like valid state.

When you see these values in a crash dump, you know immediately what kind of bug
you're looking at.

-----

*** The defense mechanisms, together
These features are complementary. Here's the full picture in one place.

**** Guard pages (=G=)
An unmapped page placed after each page-size-or-larger allocation.

Any access past the end of the allocation hits the guard page and faults.

#+begin_src sh
sysctl vm.malloc_conf='G'
#+end_src

The =<= and =>= flags adjust the free page cache size — each =<= halves it,
each =>= doubles it. Default cache size is 64 single-page allocations.

To enable guard pages and halve the cache:
#+begin_src sh
sysctl vm.malloc_conf='G<'
#+end_src

**** Junk filling (=J= / =j=)
Level 1: freed memory gets filled with =0xdf=.

Level 2: freshly allocated memory also gets filled with =0xdb= before being
handed to the caller.

=J= increments the junk level, =j= decrements it. To reach level 2:

#+begin_src sh
sysctl vm.malloc_conf='JJ'
#+end_src

**** Redzones (=R=)
Small allocations get padding around the requested size.

The canary check on free catches anything written into that padding.

Useful for off-by-one overflows that don't reach a guard page.

#+begin_src sh
sysctl vm.malloc_conf='R'
#+end_src

**** Use-after-free protection (=F=)
Freed pages get =mprotect='d to =PROT_NONE= before entering the cache.

Any dangling pointer access faults immediately — even before the page is
returned to the kernel.

#+begin_src sh
sysctl vm.malloc_conf='F'
#+end_src

**** Combining flags
#+begin_src sh
# Strong development / fuzzing setup
sysctl vm.malloc_conf='GJJRF'

# Shorthand: all security-relevant options at once
sysctl vm.malloc_conf='S'
#+end_src

For production: =GF= is a reasonable baseline. Full =GJJRF= adds measurable
overhead on allocation-heavy workloads.

-----

*** Why classic heap exploits fail here
Concrete example: the =unsafe unlink= technique against glibc's ptmalloc.

In ptmalloc, freeing a chunk triggers a doubly-linked list unlink operation on
in-band metadata. The =fd= and =bk= pointers — "forward" and "back" — sit
immediately before the user data in memory.

If you can overflow a buffer into those pointers before the free, the unlink
operation writes an attacker-controlled value to an attacker-controlled address.

That's an arbitrary write primitive. It worked for years across real software.

Here's the minimal C that creates the vulnerable condition:

#+begin_src c
/*
 * Vulnerable pattern against ptmalloc (glibc).
 * Does NOT require any glibc internals to understand —
 * the point is the predictable layout.
 */

struct Target {
    char buf[64];
    void (*fn)(void);  /* function pointer immediately after the buffer */
};

struct Target *t = malloc(sizeof(*t));
t->fn = legit_function;

/*
 * Attacker controls input. memcpy doesn't check bounds.
 * 72 bytes overwrites buf (64) plus fn (8 on 64-bit).
 */
memcpy(t->buf, attacker_input, 72);

t->fn();  /* now points to attacker-controlled address */
#+end_src

The exploit works because the layout is deterministic.

=buf= and =fn= are always at the same offset within the struct.
The struct is always at a predictable distance from other allocations.

Against otto-malloc, this attack runs into several walls at once:

1. *No predictable adjacency between allocations.*
   Small allocations go into randomly selected slots within chunk pages.
   Two consecutive =malloc= calls for the same size don't produce adjacent
   memory. The offset between =t= and anything else in the heap is not fixed.

2. *No in-band metadata to corrupt.*
   ptmalloc's =fd=/=bk= sit at =(ptr - 16)=. In otto-malloc, there's nothing
   there. =chunk_info= lives in a separate mapping. Running past the buffer
   hits either another chunk slot, a guard page, or unmapped memory.

3. *Chunk canary fires on free.*
   If the overflow crosses a chunk page boundary, the canary check in
   =chunk_info= fires on the next =free= in that bucket.
   The process aborts before the corrupted state propagates.

4. *Guard page for large allocations.*
   If =t= were page-size or larger, any write past its end hits an unmapped page
   immediately — before =t->fn= is ever called.

None of these individually make exploitation impossible.

Together, they eliminate the determinism exploitation depends on.

Heap grooming — arranging allocations in specific patterns before triggering a
bug — becomes unreliable when placement is randomized and metadata is
unreachable.

*** Comparison with other allocators
| Feature                   | OpenBSD malloc (otto) | glibc malloc           | jemalloc / tcmalloc        |
|---------------------------+-----------------------+------------------------+----------------------------|
| Heap source               | mmap only             | mixed sbrk/mmap        | custom arenas, mostly mmap |
| Metadata location         | out-of-band           | in-band                | in-band                    |
| Randomization             | high                  | limited or optional    | varies                     |
| Guard pages               | optional via config   | rarely default         | rarely default             |
| Use-after-free detection  | strong (junk + unmap) | limited                | limited                    |
| Overflow detection        | chunk canary + guard  | depends on debug mode  | not default                |
| Failure mode              | abort on inconsistency| undefined / continuing | undefined / continuing     |
| Performance priority      | safety > speed        | speed                  | speed and fragmentation    |
| Default security posture  | hardened by design    | performance-oriented   | performance-oriented       |

*** Practical notes for development
- For fuzzing and testing, =GJJRF= is the right combination.
  Bugs that are latent on glibc often become immediate crashes here.

- The junk patterns are diagnostic signals.
  =0xdb= in a crash → uninitialized read.
  =0xdf= in a crash → use-after-free.

- Guard pages and =F= both increase virtual memory usage.
  On constrained targets, pair them with =<= to reduce the free page cache.

- The source is 2700 lines and well-commented. Worth reading directly.
  [[https://github.com/openbsd/src/blob/master/lib/libc/stdlib/malloc.c][openbsd/src: lib/libc/stdlib/malloc.c]]

-----

*** What I took away from this
The design is coherent. Every decision points in the same direction.

Metadata out-of-band → can't corrupt it through a buffer overflow.

Randomized slot selection → can't predict layout, can't groom the heap reliably.

=__relro= on global config → can't corrupt the allocator's own settings.

Canaries on =dir_info= and chunk boundaries → corruption detected at the point
it happens, not ten allocations later.

Junk fill → uninitialized reads and use-after-free are immediately visible.

Guard pages → overflows into unmapped memory, immediate fault.

Fail fast on any inconsistency → the bug surfaces where it happened.

None of these are magic. Together, they add up to an allocator that treats
memory misuse as a hard contract violation rather than undefined behavior you
get to exploit later.

That's the point.

** 608.1 — Proactive Detection and Response

The course opens with something I didn't expect: a section on how to

Not just the technical side, the coordination, the communication with

**** Honeypots, honey tokens, and canaries

This was one of the sections I found most interesting.

The idea is straightforward: place things in your environment that have
no legitimate business reason to be touched. If something interacts with
them, you know immediately that something is wrong.

Honeypots are fake systems. Honey tokens are fake credentials, documents,
or data. Things an attacker would find valuable and try to use.
[[https://canarytokens.org][Canary tokens]] are a practical implementation of this: you generate a
token (a URL, a document, a DNS name), embed it somewhere, and get an
alert the moment it's triggered.

What makes this approach interesting from a detection standpoint is that
it has near-zero false positives. There is no legitimate reason for
anyone to access a canary token. When it fires, something is wrong.

The chapter concludes with threat intelligence. How to build internal
intel, and how to consume external sources. [[https://www.misp-project.org][MISP]] and [[https://filigran.io/solutions/open-cti/][OpenCTI]] are both
covered as platforms for managing and sharing threat intel.

The lab scenario throughout 608.1 involves a compromised environment at
a fictional company. A threat intel report on the adversary targeting
them is the starting point for the investigation, which continues
through the rest of the course.

-----

** 608.2 — Scaling Response and Analysis
:PROPERTIES:
:CUSTOM_ID: scaling-response-and-analysis--608-dot-2
:END:

This is where the tooling gets serious.

The challenge in enterprise IR isn't finding the right artifact on one
machine. It's collecting the right artifacts across hundreds or thousands
of machines fast enough to matter, without drowning in noise.

608.2 introduces [[https://docs.velociraptor.app/][Velociraptor]] as the primary answer to that problem.

**** Velociraptor

Velociraptor is an endpoint visibility and collection platform. You deploy
an agent to your endpoints, write queries in its own query language (VQL),
and collect forensic artifacts at scale across the entire fleet, or
targeted at specific hosts.

What makes it stand out is the depth. You're not just pulling logs.
You can query running processes, parse specific artifact types, hunt for
IOCs across every machine simultaneously, and drill into individual hosts
for deeper analysis, all from one interface.

The course also covers [[https://github.com/orlikoski/CyLR][CyLR]] for rapid triage collection, and how to
ingest that data into [[https://www.elastic.co/elasticsearch][Elasticsearch]] for fast searching and aggregation.

**** Timesketch

Once you have data, you need to make sense of it. That's where
[[https://timesketch.org/][Timesketch]] comes in.

Timesketch is a platform for collaborative timeline analysis. You load
forensic artifacts event logs, filesystem timestamps, network data 
and it builds a searchable, filterable timeline across all of it.

The collaborative part matters in enterprise IR. Multiple analysts can
work the same timeline simultaneously, add annotations, tag events, and
build a shared picture of what happened and when.

Working through the lab scenario in Timesketch was the moment the course
clicked for me. You go from a pile of artifacts to a coherent sequence of
attacker actions. The timeline makes the narrative visible.

Velociraptor and Timesketch individually are both powerful tools.
Integrated, Velociraptor collecting at scale, Timesketch making the
data navigable, they cover a large part of what enterprise IR actually
requires.

The chapter also covers EDR data from tools like [[https://learn.microsoft.com/de-de/sysinternals/downloads/sysmon][Sysmon]], and importantly,
common techniques attackers use to bypass or blind EDR tooling. Knowing
the evasion techniques is as important as knowing the detection ones.

** 608.3 — Modern Attacks against Windows and Linux
:PROPERTIES:
:CUSTOM_ID: modern-attacks-against-windows-and-linux-dfir--608-dot-3
:END:

608.3 moves from the tooling layer to the artifact layer.

The focus here is on what attackers actually do on Windows and Linux
systems, and what traces they leave behind.

**** Windows: ransomware and living off the land

The course covers ransomware from an IR perspective, not how it works
cryptographically, but what artifacts it leaves and how to reconstruct
the timeline of a ransomware incident.

More interesting to me was the [[https://lolbas-project.github.io/#][Living Off the Land]] (LOTL) section.

LOTL attacks use built-in Windows binaries and scripting capabilities
to do malicious things, =certutil= for downloads, =wmic= for lateral
movement, =mshta= for execution. No custom malware to detect, no
suspicious executables to flag. Just Windows doing what Windows does,
pointed in the wrong direction.

[[https://github.com/SigmaHQ/sigma][Sigma]] rules get more attention here as a detection layer, a way to
write detection logic in a vendor-neutral format that can be translated
to whatever SIEM or detection platform you're running.

**** Linux DFIR

The Linux section starts with common attack vectors and then covers the
fundamentals of forensic analysis on Linux systems, differences between
distributions, filesystem considerations, initial triage approach, and
deeper artifact analysis.

It concludes with hardening and logging recommendations that would
actually make future investigations easier. That's a useful framing:
good logging isn't just compliance, it's evidence preservation.

** 608.4 — macOS and Docker Containers
:PROPERTIES:
:CUSTOM_ID: analyzing-macos-and-docker-containers--608-dot-4
:END:

**** macOS

The macOS section covers the Apple Filesystem (APFS) and the specific
artifacts that matter for IR on macOS, property list (plist) files,
log formats, acquisition approaches.

macOS has its own challenges for IR: Apple's privacy controls affect
what you can collect, and the forensic tooling ecosystem is narrower
than on Windows. The course is honest about that.

**** Docker containers

The container section was one I hadn't seen covered well elsewhere.

The approach is a specific triage workflow: how to assess a running
container quickly, what artifacts are available at the container level
versus the host level, and how the architecture of Docker affects what
evidence exists and where.

Container forensics is a different mental model from host forensics.
The container might be long gone by the time you're investigating.
Understanding where the evidence persists, in image layers, in host
logs, in orchestration tooling, is the core skill here.

** 608.5 — Cloud Attacks and Response
:PROPERTIES:
:CUSTOM_ID: cloud-attacks-and-response--608-dot-5
:END:

The final technical chapter covers IR in Microsoft Azure / M365 and AWS.

**** Microsoft 365 and Azure

The M365 section is heavily focused on log analysis, specifically the
Unified Audit Log, which is the primary source of truth for what happened
in an M365 environment. Suspicious logon patterns, email forwarding rules,
OAuth application grants, these are the artifacts that matter in M365
incidents, and the course walks through how to find and interpret them.

The [[https://attack.mitre.org/matrices/enterprise/cloud/][MITRE ATT&CK Cloud Matrix]] is used as a framework throughout,
which helps map observed activity to known attacker techniques.

Coverage includes Entra ID (formerly Azure AD), Exchange, SharePoint,
and Teams, the services that appear in most real M365 incidents.

**** AWS

The AWS section covers the general architecture and the specific logs
and services that matter for IR: CloudTrail, GuardDuty, VPC Flow Logs,
S3 access logs.

What I found useful here was the discussion of architecting for
response, setting up a dedicated security account as an isolated enclave,
using AMIs as analysis templates, and automating IR tasks with Lambda
and Step Functions. The idea is to design your AWS environment so that
incident response is faster and more reliable before an incident happens.

set idfile /var/monit/id

#+end_src

program outputs, network I/O, and service start/stop/restart

#+begin_src conf
set limits {
     programOutput:     512 B,
     sendExpectBuffer:  256 B,
     fileContentBuffer: 512 B,
     httpContentBuffer: 1 MB,
     networkTimeout:    5 seconds
     programTimeout:    300 seconds
     stopTimeout:       30 seconds
     startTimeout:      30 seconds
     restartTimeout:    30 seconds
}

set mailserver localhost

 slots 200
set mail-format { from: root@monit }

** Introduction
If you're running Elasticsearch on a single node — like a Raspberry Pi or small lab setup like I am —
you might notice some indices appear with a =yellow= health status.

This show article explains what that means and how to fix it, especially in resource-constrained, *single-node* environments.

** What Does "Yellow" Mean?
In Elasticsearch:

- =green=: All primary and replica shards are assigned and active.
- =yellow=: All primary shards are active, but at least one *replica shard* is unassigned.
- =red=: At least one *primary shard* is missing → critical!

** Why Yellow Happens on Single Nodes
In single-node clusters, Elasticsearch cannot assign replica shards (because replicas must be on a different node).  
So any index with replicas will *always* be yellow unless:

- You add more nodes (not ideal on a Raspberry Pi)
- Or: You disable replicas (=number_of_replicas: 0=)

** Step-by-Step: Diagnose Yellow Shards

*** 1. List all yellow indices
#+begin_src sh
GET _cat/indices?v&health=yellow
#+end_src

*** 2. See why a shard is unassigned
#+begin_src sh
GET _cluster/allocation/explain
#+end_src

*** 3. Inspect shard assignment of a specific index
#+begin_src sh
GET _cat/shards/.monitoring-beats-7-2025.08.06?v
#+end_src

Example output:

#+begin_example
index                              shard prirep state      docs store ip        node
.monitoring-beats-7-2025.08.06     0     p      STARTED    7790 5.9mb 127.0.0.1 mynode
.monitoring-beats-7-2025.08.06     0     r      UNASSIGNED
#+end_example

→ The =r= (replica) is unassigned → =yellow= status.

** How to Fix It

*** A. Fix an individual index
Set replicas to zero:

#+begin_src sh
PUT .monitoring-beats-7-2025.08.06/_settings
{
  "index" : {
    "number_of_replicas" : 0
  }
}
#+end_src

This changes the index health from =yellow= to =green=.

*** B. Automatically fix all yellow indices
If you want to *automate* the fix, use this (Kibana Dev Tools):

#+begin_src js
GET _cat/indices?health=yellow&format=json
#+end_src

Then for each index in the result:

#+begin_src js
POST <your_index>/_settings
{
  "index": {
    "number_of_replicas": 0
  }
}
#+end_src

*** C. Prevent future yellow indices
Disable replicas by default using an index template:

#+begin_src json
PUT _template/no-replica-default
{
  "index_patterns": ["*"],
  "settings": {
    "number_of_replicas": 0
  }
}
#+end_src

> ⚠️ This applies to *all* future indices. _Only do this in single-node environments_.

** Conclusion
Yellow indices aren't dangerous by default — they just mean you're missing redundancy.  
In small environments, it's perfectly safe to run with zero replicas.

Just don't forget to:
- Monitor your shard health
- Disable replicas if you only have one node
- Automate where you can

** Introduction
So I had this USB Disk attached to my OpenBSD Router used as storage, one saturday when I was walking by
I noticed the weird clicking sounds from the disk. So I knew my time was running before the disc would fail.

Curiously, when I plugged the same drive into a Linux box, it *was* detected and even
showed a valid OpenBSD partition table. That gave me a glimmer of hope:
maybe the hardware wasn’t completely dead yet.

So, for fun (and a little bit of stubborn curiosity), I decided to spend
the weekend seeing how much I could rescue from it.

This post documents the process part forensic experiment, part recovery attempt,
and part “let’s see what happens if I do this.”

** Phase 1: Identifying the Disk under Linux

Before doing anything risky, I wanted to be sure I was imaging the *right* disk.  
The idea was to identify the OpenBSD partition and dump it to an image file.

*** Listing block devices

#+begin_src sh
lsblk -o NAME,SIZE,FSTYPE,TYPE,LABEL,UUID
#+end_src

That gives a good overview which disks are present, how large they are, and what filesystems they contain.  
Sure enough, my external USB drive showed up as /dev/sda.

*** Inspecting partition table

#+begin_src sh
sudo fdisk -l /dev/sda
#+end_src

Example output:

#+begin_example
Disk /dev/sda: 931.5 GiB, 1000204883968 bytes, 1953525164 sectors
Disk model: External USB 3.0
Sector size: 512 bytes
Disklabel type: dos
Device     Boot Start        End    Sectors   Size Id Type
/dev/sda4  *       64 1953525163 1953525100 931.5G a6 OpenBSD
#+end_example

Perfect. The OpenBSD partition was still there (/dev/sda4), and it even reported the correct size.

- The *Start sector* (64) is important later for offset calculations.
- Type *a6 OpenBSD* confirmed the filesystem was OpenBSD-specific (likely softraid).
- Knowing the *sector size* (512 bytes) ensured that later tools like *dd* or *ddrescue* wouldn’t misalign reads.

At this point, the goal was to make a *bit-for-bit copy* of that partition, compress it, and work
on the image rather than risk further damage to the actual disk.

** Phase 2: Creating a Compressed Disk Image

For imaging, I decided to use GNU ~ddrescue~ it’s great for flaky disks and can retry sectors intelligently.

*** Installing ddrescue

On Fedora, installation was trivial:

#+begin_src sh
sudo dnf install ddrescue
#+end_src

*** First Attempt (Quick and Dirty)

I tried a fast, one-shot dump not ideal for a failing disk, but I wanted to see if it would work at all:

#+begin_src sh
sudo ddrescue -d -r3 /dev/sda4 - - | xz -T0 -c > openbsd_sda4.img.xz
#+end_src

That command streams data directly from the device, compresses it with ~xz~, and writes the result.  
It works *if the disk is healthy*. Mine wasn’t, so it failed partway through.

*** Second Attempt (Proper Forensic Mode)

So I switched to the safer, resumable method:

#+begin_src sh
sudo ddrescue -d -r3 /dev/sda4 openbsd_sda4.img openbsd_sda4.log
xz -T0 openbsd_sda4.img
sha256sum openbsd_sda4.img > openbsd_sda4.img.sha256
#+end_src

This time, ddrescue created a detailed log file so I could resume later if the system froze or the disk disconnected.  
It took most of the night, but eventually I had a clean (or mostly clean) image.

_Explanation of parameters_

- ~-r3~ retries each bad block 3 times  
- ~-d~ enables direct disk I/O  
- The *.log* file lets you stop and restart without losing progress  
- ~xz -T0~ uses all CPU cores for compression  

After the dump, I verified the output:

#+begin_src sh
ls -lh openbsd_sda4.img.xz
xz -t openbsd_sda4.img.xz   # test integrity
sha256sum openbsd_sda4.img.xz > openbsd_sda4.img.xz.sha256
#+end_src

Everything checked out a ~450 GB compressed image file safely sitting on my main system.

** Phase 3: Simulating Disk Damage (For Fun and Testing)

Since the real disk was unstable, I wanted a safe way to experiment.  
So I created a *copy of the image* and simulated damage to practice recovery techniques.

*** Creating the test image

#+begin_src sh
sudo dd if=/dev/sda4 of=openbsd_sda4.img bs=4M status=progress
#+end_src

*** Simulating corruption

To emulate bad sectors:

#+begin_src sh
dd if=/dev/zero of=openbsd_sda4.img bs=512 count=10 seek=1000 conv=notrunc
#+end_src

Now the image contained 10 intentionally corrupted sectors perfect for testing.

*** Recovering from the damaged image

#+begin_src sh
ddrescue -d -r3 openbsd_sda4.img openbsd_sda4_recovered.img openbsd_sda4_recovery.log
#+end_src

And just like that, I was able to practice recovery without touching the actual hardware again.

*** Optional Compression

#+begin_src sh
xz -T0 openbsd_sda4.img
#+end_src

It’s amazing how much you can still do with raw disk images and a few tools.

** Phase 4: Performance Tuning and System Stability

During the rescue, I learned (the hard way) that ~ddrescue~ can saturate I/O and make your system lag like crazy.  
So I ended up using this combination for a gentler approach:

#+begin_src sh
sudo ionice -c2 -n7 nice -n19 ddrescue -b 4096 -B 4096 /dev/sda4 openbsd_sda4.img
#+end_src

And, for long operations, running it inside ~tmux~:

#+begin_src sh
tmux new-session -s rescue
sudo ddrescue -d -r3 /dev/sda4 openbsd_sda4.img openbsd_sda4.log
# Detach with Ctrl-B D
#+end_src

Later, I could simply:

#+begin_src sh
tmux attach -t rescue
#+end_src

That setup saved me more than once when I accidentally closed an SSH session.

** Phase 5: Next Steps — Future Analysis

Once I had a full image, the plan was to:

1. Decompress it (~unxz openbsd_sda4.img.xz~)
2. Attach it as a loopback device under Linux, or use ~vnconfig~ under OpenBSD
3. Attempt to reassemble the ~softraid~ volume using ~bioctl~
4. If all goes well — mount the decrypted filesystem and access my old data

That’s a topic for another weekend. But getting to this
point already felt like a small victory.

** Conclusion

What started as a “let’s see if I can still read this disk” experiment turned into
a proper mini-forensics exercise. Even though the original USB drive was dying,
I managed to preserve most of its data and learned a ton in the process.

Allover it was quite fun to do something forensics related on a OpenBSD target, I guess it is
something you don't come across everyday but when you do its good to be prepared I think.

Key takeaways:

- ~ddrescue~ is your friend for unstable media  
- Always work on *images*, not the original device  
- Compression and checksums are cheap insurance  
- And most importantly: never underestimate what you can recover with a bit of patience

Not a bad way to spend a weekend. Nevertheless I would like to find a purely OpenBSD Based solution.
But I was not able to find the dd_rescue in the ports and packages of OpenBSD. If someone knows how
to do this on purely OpenBSD please contact me.

** Why GPG?
In an age where digital identities are easily faked and impersonation is just a few clicks away, I decided to take a step forward in securing mine. GPG (GNU Privacy Guard) provides a robust way to authenticate, encrypt, and sign digital content. In this post, I’ll walk you through how I:

- Created a GPG key pair
- Set up subkeys and stored them on my YubiKey
- Published my public key on my website
- Signed and encrypted personal documents for secure public sharing
- Configured email signing using GPG

** Step 1: Installing GPG
To start, I made sure GPG was installed. Here's how I did it on each of my systems:

**On Ubuntu/Debian:**
#+BEGIN_SRC shell
sudo apt update && sudo apt install gnupg
#+END_SRC

**On Fedora 40:**
#+BEGIN_SRC shell
sudo dnf install gnupg2
#+END_SRC

**On OpenBSD 7.6:**
#+BEGIN_SRC shell
doas pkg_add gnupg
#+END_SRC

Check your installation:
#+BEGIN_SRC shell
gpg --version
#+END_SRC

** Step 2: Creating My GPG Key Pair
I created a new key using:

#+BEGIN_SRC shell
gpg --full-generate-key
#+END_SRC

Here’s what I chose:

- Key type: ~ed25519~ (modern and compact) or ~RSA and RSA~ (widely compatible)
- Key length: 4096 bits (if RSA)
- Expiration: 2 years (I can always renew)
- My real name or handle
- My preferred contact email
- A strong passphrase, saved in a password manager

After generating the key, I listed it and saved the fingerprint:

** Step 3: Creating Subkeys and Moving Them to My YubiKey
I created subkeys for:
- Signing
- Encryption
- Authentication

Then, I moved the subkeys to my YubiKey using GPG’s interactive editor:

#+BEGIN_SRC shell
gpg --edit-key Dirk.L
gpg> addkey <- once for signing, engryption, auth 
gpg> keytocard
gpg> save
#+END_SRC

⚠️ *Be cautious:* Once moved to the YubiKey, the subkey *no longer exists* on disk.

More guidance: [[https://developers.yubico.com/PGP/][YubiKey + GPG official instructions]]

** Step 4: Publishing My Public Key
I exported my key in ASCII format so others could import it easily:

#+BEGIN_SRC shell
gpg --export --armor you@example.com > publickey.asc
#+END_SRC

I uploaded ~publickey.asc~ to my website and linked it like this:

#+BEGIN_SRC html
<a href="/publickey.asc">🔑 Download my GPG public key</a>
#+END_SRC

Additionally, I displayed my key’s fingerprint on the page so that people can verify its authenticity manually.

** Step 5: Email Signing and Encryption
I configured email signing using my GPG key.

**For Thunderbird (Linux, OpenBSD, Windows):**
- OpenPGP support is built-in.
- I enabled signing for all outgoing mail.
- The key lives on the YubiKey, so no key is stored on disk.

**For Mutt / CLI mailers:**
- I used `gpg-agent` for passphrase and key handling.
- Configured ~.muttrc~ to sign and/or encrypt automatically.

Signing ensures message authenticity. If recipients have my key, they can encrypt replies.

** Step 6: Signing and Encrypting Documents for the Public
To safely share personal certificates and private files, I signed and optionally encrypted them:

#+BEGIN_SRC shell
# Sign only (adds signature block)
gpg --sign --armor diploma.pdf

# Sign and encrypt with a password (no public key needed)
gpg --symmetric --armor --cipher-algo AES256 diploma.pdf
#+END_SRC

This way, the document is verifiably mine and only decryptable with the shared password.

The encrypted ~.asc~ files can be uploaded to the website, with instructions for downloading and decrypting.

** Step 7: Offline Backup of My Master Key
Before moving entirely to the YubiKey, I backed up the master key offline:

#+BEGIN_SRC shell
gpg --export-secret-keys --armor > masterkey-backup.asc
#+END_SRC

I stored it on an encrypted USB drive with either one:
- **LUKS** (on Linux)
- **OpenBSD softraid(4)** encryption

Rolling out GPG was super easy. With my identity cryptographically verifiable, email
signing in place, and secure document sharing live on my site, I now
have a strong, decentralized identity system.
